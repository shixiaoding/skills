# 资料收集总结 Skill

## 职责

围绕用户指定主题，完成：搜索资料 → 分类整理 → 生成 Markdown 文档 → 输出摘要；仅在用户明确要求时同步 Notion。

同时管理版本策略：新建版本、修改当前文档、或仅更新资料库。

### 外部写入边界

Notion 同步是可选操作。只有用户明确要求同步且 Notion MCP 已连接时才执行；未要求同步或服务不可用时，保留 Markdown 输出并说明未同步原因。

---

## 总原则

### 默认不覆盖旧文档

除非用户明确说"修改当前文档"、"直接改这份"、"覆盖旧内容"，否则创建新版本。

### 资料库优先完整性

当用户强调"资料要全"、"不要丢链接"、"保留历史资料"时，正文可以精简，资料库表格不能随意删减。

### 待验证资料不删除

无法验证的资料保留但标记状态：待验证、找不到正式链接、失效原链接、可能无效、待补链、待补公众号原文。

### 修改与更新区分

| 用户意图 | 动作 |
|---|---|
| 修改 | 修改当前文档，不创建新版本 |
| 新版本 | 基于旧版演进，创建新版本 |
| 资料库更新 | 不改结构，只更新表格 |
| 定期更新 | 周期性补最新资料，只维护资料库 |

## 意图识别与执行流程

### 模式 A：首次资料收集

触发语义：帮我收集... / 帮我整理... / 生成资料库... / 调研一下...

1. **确认范围**：向用户确认主题、目标、范围、排除范围、资料深度。
2. **收集资料**：按下方「资料收集」章节执行搜索。
3. **生成文档**：按下方「文档生成」章节，使用 `assets/template.md` 作为文档骨架生成 Markdown。
4. **按需同步 Notion**：用户明确要求且服务可用时，按下方「Notion 同步」章节创建新页面。
5. **输出摘要**：按「输出摘要格式」输出结果。

### 模式 B：新版本生成

触发语义：生成 v2/v3 / 基于上一版继续 / 不要覆盖旧文档 / 重新生成一版

1. **确定版本**：版本号递增（v1→v2→v3），保留旧版本。命名规则见 `references/specs.md`。
2. **补充资料**：按「资料收集」章节搜索新增资料，非全量重搜。
3. **生成新版**：在旧版基础上按「文档生成」章节生成新版 Markdown。
4. **按需同步 Notion**：用户明确要求且服务可用时创建新页面（保留旧页面）。
5. **输出版本差异摘要**：按新增/调整/降级/待验证四类输出。

### 模式 C：修改当前文档

触发语义：修改当前文档 / 直接改这份 / 这章重写一下 / 总结不是我想要的

1. **明确范围**：确认修改哪些章节。
2. **定点修改**：只改用户指出的位置，其他章节不动。
3. **按需同步 Notion**：用户明确要求且服务可用时更新对应页面。
4. **输出修改摘要**：修改章节 + 修改原因 + 保留内容。

### 模式 D：资料库更新

触发语义：这版确认了 / 后续只做资料库更新 / 补充最新资料 / 更新链接状态

1. **搜新资料**：按「资料收集」章节搜索近期新增。
2. **不改结构**：只更新资料表（新增行、更新链接状态、标记失效）。
3. **按需同步 Notion**：用户明确要求且服务可用时只更新页面的资料表部分。
4. **输出更新摘要**：新增资料 + 链接状态变化 + 待补链。

### 正式版合并

触发语义：其他文档不要了 / 合并成一个正式版 / 以这个为准

1. 将分散补充页的有效内容合并到主文档。
2. 主文档声明"以本页为准"。
3. 如工具支持，归档或重命名旧页面。

## 资料收集

### 搜索步骤

按优先级依次搜索，来源优先级见 `references/specs.md`。

**1. 官方优先：**
```
主题 + official docs / GitHub / documentation / guide
```

**2. 大厂实践：**
```
主题 + 腾讯 / 阿里 / 字节 / 得物 / 美团 / 技术实践 / AI Coding
```

**3. GitHub / 开源：**
```
主题 + GitHub / awesome / template / examples / 中文 / docs zh
```

**4. 论文：**
```
主题 + arXiv / paper / empirical study / agentic workflow
```

### 资料分类

按以下分类输出：
- 官方资料 / 大厂实践 / 国内社区 / 国外资料
- OpenAI / Codex / Claude Code
- GitHub / Awesome / 论文
- 待验证资料 / 失效链接

### 输出表格

每条资料必须包含以下字段，字段定义见 `references/specs.md`：

| 分类 | 资料名称 | 来源 | 类型 | 可信度 | 推荐程度 | 主要价值 | 链接 | 可访问性 | 是否原文 | 替代链接 | 状态 | 备注 |

### 失效资料处理

1. 保留标题。
2. 标记"失效原链接"或"找不到正式链接"。
3. 搜索替代链接，填入替代链接列。
4. 不把失效资料写成确定结论。

### 收集完成标准

- 覆盖官方资料、大厂实践、开源社区、GitHub 项目、待验证资料
- 每条链接有可访问性状态
- 每条资料有可信度分级
- 输出了资料清单

## 文档生成

### 写作原则

- **正文精简，资料库完整**：正文讲清主线，资料表保留完整链接，不能为可读性删除资料。
- **主题是中心**：文档围绕主题，不让工具或方法论喧宾夺主。
- **不确定资料保留但标记**：不删除待验证/失效/可能无效的资料。

### 文档结构

使用 `assets/template.md` 作为文档骨架，按以下章节组织：

文档定位 → 主题是什么 → 核心概念和边界 → 学习路径 → 官方资料库 → 大厂实践资料库 → 国内社区实践资料库 → 国外资料库 → 工具链/生态资料库 → GitHub/开源社区资料库 → 待验证资料 → 采用策略 → 后续维护规则

可根据主题调整章节，但不能删除"资料库"和"待验证资料"章节。

### 章节写法

**概念解释**：白话、简洁、不空泛、有场景映射。

**实践案例**使用结构：
```
### 公司/项目实践
实践重点：
适合学习：
实践映射：
| 外部做法 | 我可以怎么用 | 是否采用 | 备注 |
资料表：
| 资料 | 推荐程度 | 可信度 | 链接 | 可访问性 | 状态 |
```
实践映射表和资料链接表必须分开。

### 输出格式

- Markdown，标题层级清晰，表格字段统一
- 不过度使用"本章结论"，用"我的理解"、"我会怎么用"、"需要注意"
- 长资料库以表格为主
- 不把待验证资料写成确定结论
- 适合 Notion 阅读，可后续维护

## Notion 同步

仅当用户明确要求同步时，使用已连接的 Notion MCP 服务。同步前确认服务可用；不可用时不阻塞资料整理，改为输出 Markdown 并说明原因。

详细规则见 `references/specs.md` Notion 同步规则部分。

### 创建新页面

适用：首次生成、新版本、"不要覆盖旧文档"。

1. 按 `references/specs.md` 目录选择规则确定目标目录。
2. 调用 notion MCP 创建页面，写入 Markdown。
3. 返回 Notion 链接。

### 更新当前页面

适用：用户明确说"修改当前文档"、"直接改这份"、"覆盖旧内容"。

1. 定位指定页面，只修改指定章节。
2. 返回修改摘要（修改位置 + 保留内容）。

### 资料库更新

适用："这版确认了"、"以后只做资料库更新"。

1. 不改主结构，只更新资料表（新增行、更新链接状态、补充替代链接）。
2. 返回更新摘要。

### 页面命名

- 首次：`主题资料库`
- 新版本：`主题资料库 v1` / `主题资料库 v2`
- 日期版本：`主题资料库 2026-05-21`

### 更新保护

默认不覆盖旧文档。覆盖前必须满足：用户明确说"修改当前文档"、"覆盖旧内容"、或指定了具体页面。不满足则创建新版本。

---

## 防误覆盖规则

**必须新建版本的情况：**
- 用户说"不要覆盖"、"生成新版"
- 用户未明确允许修改旧文档
- 旧文档已被确认过
- 本次是大范围结构重组

**可以修改旧文档的情况：**
- 用户明确说"修改当前文档"
- 用户指出具体章节需修改
- 用户说"其他文档不要了，更新主文档"
- 要求合并为唯一正式版

```
- Notion 链接（如已同步）：...
- 新增资料数量：...
- 待验证资料数量：...
- 失效链接数量：...

## 重点变化
- ...

## 后续建议
- ...
```

### 修改当前文档

```
## 修改摘要
- 修改章节：...
- 修改原因：...
- 保留内容：...
```

### 资料库更新

```
## 资料库更新摘要
- 新增资料：...
- 链接状态变化：...
- 待补链：...
- 建议后续补充：...
```
