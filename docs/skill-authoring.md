# Skill 编写约定

## 目标

一个 Skill 应把已经验证过的高频工作方法变成可复用能力，而不是记录一次性提示词或尚未验证的想法。

## 最小结构

```text
skills/<skill-name>/
├── SKILL.md                 # 必需：定义能力与执行边界
├── assets/                  # 可选：模板、静态资源
└── references/              # 可选：字段规则、来源优先级等稳定参考
```

`SKILL.md` 必须保留 YAML front matter 中的 `name` 和 `description`。目录名、`name` 与文档标题应使用同一语义，避免 Agent 无法判断该调用哪个 Skill。

## npm 安装器兼容

`@helloxiaoding/ding-skills` 会扫描 `skills/*/SKILL.md` 自动发现可安装 Skill。新增 Skill 时，目录名必须与 front matter 的 `name` 完全一致，并保留非空的 `description`；否则 `ding-skills list` 和安装命令会拒绝继续，避免发布不可识别的内容。

Skill 附属文件可放在 `agents/`、`assets/`、`references/` 等目录中。安装器会随 Skill 目录完整复制这些文件，不需要为 npm 分发维护第二份内容。

## 编写要求

1. **触发条件清晰**：描述用户意图，而不是只列几个关键词。
2. **输入与依赖明确**：区分必需输入和可选服务；外部写入（如 Notion）必须由用户明确请求。
3. **步骤可执行**：说明关键决策点、失败处理和完成标准。
4. **输出可验证**：给出结果格式、保存位置或验收标准。
5. **范围保持小**：一个 Skill 只解决一类问题；复杂流程可组合多个 Skill，但不要堆成万能 Skill。

## 新增前检查

新增前先回答：

- 是否已经在真实任务中使用并验证？
- 是否与现有 Skill 重复？
- 是否能脱离个人隐私、私有路径和特定项目上下文运行？
- 是否需要外部写入或付费服务？如果需要，是否已说明授权与降级方案？

四项中任一项无法回答清楚时，先保留为草稿或案例，不新增为正式 Skill。

## 发布与维护

- 已发布、会影响使用者的变更记录在根目录 [Changelog](../CHANGELOG.md)。
- 尚未验证的方向先放入 [Roadmap](roadmap.md)，不要提前建立正式 Skill 目录。
- 新增完整能力通常发布为次版本，例如 `0.2.0`；修复安装、文档或兼容性问题通常发布为补丁版本，例如 `0.1.1`。
