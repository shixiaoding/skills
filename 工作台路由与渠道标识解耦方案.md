# 工作台路由与渠道标识解耦方案

## 1. 背景

当前渠道侧存在多个 App 与多套业务身份并存的现状：

- RS：锐锢销售，对应直销入口
- RP：锐销宝，对应企拍入口
- RD：销配，对应司机入口

现网 CRM 网关接入规范中，`X-Display-ChannelCode` / `channelCode` 的语义是**请求来源通道**，主要用于网关责任链路由与鉴权链路分发，例如 RS / RP / RD 的责任链选择。

但在“渠道多角色改造二期”中，已经出现以下新诉求：

1. 同样从 **RS App** 进入的用户，可能需要根据其组织身份进入不同工作台：
   - 直销体系 → RS 工作台
   - 品牌体系 → RSO 工作台
   - 市场体系 → RDT 工作台
2. 同一个手机号可能同时命中多个架构身份，需要支持**角色选择页**与**身份切换**
3. 门店、订单、售后单等业务对象的“渠道类型”判定规则，已经不再等同于入口通道，而是基于门店关联人员/组织关系计算得出

因此，现有 `channelCode` 已无法同时承担以下三类职责：

- 入口来源识别
- 当前工作台识别
- 业务对象渠道归类

如果继续混用，会导致：
- 同一 `channelCode=RS` 下无法区分直销工作台 / 品牌工作台 / 市场工作台
- 用户切换身份后，下游无法准确识别“当前所处工作台”
- 门店/订单对象的业务归类与用户当前入口含义混淆
- 后续 RS / RP / RD 能力整合时，模型演进困难

---

## 2. 现状问题

### 2.1 `channelCode` 语义过载

当前 `channelCode` 实际表达的是“请求从哪个入口进入”：
- RS
- RP
- RD

它适合用于：
- 网关责任链路由
- 入口来源识别
- 老系统兼容
- 版本/终端埋点统计

但不适合用于：
- 工作台匹配
- 菜单加载
- 首页指标切换
- 数据权限视角切换

### 2.2 工作台判定与入口通道不再一一对应

在多角色二期中，用户即使从 RS App 登录，也可能进入：
- 直销工作台
- 品牌工作台
- 市场工作台

也就是说：

`入口通道 != 当前工作台`

### 2.3 业务对象渠道归类已有独立规则

门店渠道类型、订单渠道类型、售后单渠道类型等，已经有独立的业务归类规则，例如：

- 品牌体系人员关联的门店，业务归类可能仍记为“直销”
- 市场体系人员关联的门店，业务归类可能仍记为“销配”

因此：

`业务对象渠道归类 != 当前工作台 != 请求入口通道`

### 2.4 后续统一 App 场景下扩展性不足

后续若 RS / RP / RD 逐步整合为统一渠道 App，入口通道将趋于收敛，但工作台身份仍会长期存在。  
若当前不做解耦，未来将面临更大范围的字段语义重构。

---

## 3. 设计目标

本方案的目标如下：

1. **保留 `channelCode` 作为入口通道标识**
2. **新增 `workspaceCode` 作为当前工作台标识**
3. **保留/新增 `bizChannelType` 作为业务对象归类标识**
4. 以**组织根节点**作为工作台匹配的核心来源
5. 支持单身份直达、多身份选择、身份切换、默认身份记忆
6. 兼容现有 RS / RP / RD 网关、鉴权、UserInfo 透传机制
7. 为后续统一 App、多角色扩展、菜单动态配置提供基础能力

---

## 4. 设计原则

### 4.1 单一职责原则

每个字段只表达一个核心语义：

- `channelCode`：请求入口来源
- `workspaceCode`：当前工作台身份
- `bizChannelType`：业务对象渠道归类

禁止一字段多义。

### 4.2 组织驱动原则

工作台身份由**用户所在组织树的根节点**决定，而不是由入口 App 决定。

### 4.3 向后兼容原则

不推翻现有网关与 Header 责任链，仅在现有体系上新增工作台上下文。

### 4.4 渐进演进原则

优先完成“概念解耦 + 路由可用”，后续逐步推进菜单、权限、数据、对象归类等全面切换。

---

## 5. 核心概念与字段模型

### 5.1 概念划分

#### 5.1.1 入口通道（Channel）

表示请求从哪个客户端/入口发起。

示例：
- RS
- RP
- RD

#### 5.1.2 工作台（Workspace）

表示当前用户进入了哪个业务工作台，决定：
- 首页
- 菜单
- 指标
- 数据权限口径
- 模块可见性

示例：
- RS：直销工作台
- RSO：品牌工作台
- RDT：市场工作台
- RD：销配工作台
- RP：企拍工作台

#### 5.1.3 业务归类渠道（Biz Channel Type）

表示门店、订单、售后单等业务对象的归类结果，服务于：
- 报表
- 列表
- 快照
- 数据平台口径
- 跨系统同步

示例：
- 直销
- 销配
- 企拍

### 5.2 字段建议

| 字段 | 含义 | 来源 | 使用范围 |
|---|---|---|---|
| `channelCode` | 请求入口通道 | Header / 网关 | 网关责任链、埋点、兼容 |
| `clientId` | 终端标识 | Header | 客户端识别、Passport增强 |
| `workspaceCode` | 当前工作台标识 | 登录后计算 / 用户选择 | 页面路由、菜单、权限、指标 |
| `rootOrgType` | 当前工作台对应的组织根节点类型 | 组织树映射 | 路由判定、权限辅助 |
| `bizChannelType` | 业务对象归类渠道 | 对象规则计算 | 门店/订单/售后/报表 |

---

## 6. 解耦后的整体模型

建议形成“三套码并存”的模型：

### 6.1 通道码 `channelCode`
用于：
- 网关责任链
- 鉴权链路
- 老系统兼容
- 入口来源识别

取值建议：
- RS
- RP
- RD

### 6.2 工作台码 `workspaceCode`
用于：
- 登录后路由
- 多身份切换
- 菜单加载
- 首页指标
- 模块权限
- 数据范围口径切换

取值建议：
- RS：直销工作台
- RSO：品牌工作台
- RDT：市场工作台
- RD：销配工作台
- RP：企拍工作台

### 6.3 业务归类码 `bizChannelType`
用于：
- 门店渠道类型
- 订单渠道类型
- 售后单渠道类型
- 数据平台汇总口径
- 跨系统同步

取值建议：
- DIRECT_SALES（直销）
- SALES_DELIVERY（销配）
- QIPAI（企拍）

---

## 7. 工作台匹配规则设计

### 7.1 基本思路

工作台不再通过 `channelCode` 推导，而是通过**用户所属组织节点对应的根节点**推导。

#### 规则过程

1. 用户登录成功
2. 根据用户当前有效组织归属，查出其所属根节点集合
3. 根节点映射为可用工作台列表
4. 若只有一个工作台，则直接进入
5. 若有多个工作台，则进入角色选择页
6. 用户选中后，将 `workspaceCode` 写入当前会话上下文
7. 后续页面、菜单、数据权限均按 `workspaceCode` 生效

### 7.2 根节点与工作台映射建议

| 根节点类型 | 根节点名称 | 工作台编码 | 工作台名称 | 业务归类渠道 |
|---|---|---|---|---|
| DIRECT | 直销体系 | RS | 直销工作台 | 直销 |
| BRAND | 品牌体系 | RSO | 品牌工作台 | 直销 |
| MARKET | 市场体系 | RDT | 市场工作台 | 销配 |
| DELIVERY | 销配体系 | RD | 销配工作台 | 销配 |
| QIPAI | 企拍体系 | RP | 企拍工作台 | 企拍 |

说明：

- “工作台编码”决定用户端展示与交互
- “业务归类渠道”决定门店/订单对象的默认归口倾向
- 两者可不相同，例如：
  - 品牌工作台 → 业务归类可仍视为直销
  - 市场工作台 → 业务归类可仍视为销配

### 7.3 默认工作台选择规则

#### 方案建议

1. **仅一个工作台**
   - 直接进入

2. **多个工作台**
   - 优先读取“上次退出时所选工作台”
   - 若无历史记录，则按系统默认优先级推荐
   - 同时展示角色选择页供用户切换

#### 默认优先级建议

建议以用户长期使用频次 + 业务主链路为依据，初版可配置为：

1. 直销工作台
2. 品牌工作台
3. 市场工作台
4. 销配工作台
5. 企拍工作台

最终以产品确认为准。

### 7.4 身份切换规则

用户在工作台间切换时，需要同步切换以下内容：

- 菜单
- 首页金刚位
- 工作盘点指标
- 业绩查询口径
- 排行榜口径
- 门店列表范围
- 工单 / 订单 / 售后可见范围

切换后应更新当前会话中的：
- `workspaceCode`
- `rootOrgType`
- 可选的 `groupId` / `accessibleScopes` 视角

## 8. Header 与上下文扩展建议

### 8.1 保留现有 Header

现有 Header 保持不动：
- `Authorization`
- `X-Display-ChannelCode`
- `X-Client-Id`
- `X-UserInfo-UserId`
- 其它已有上下文字段

### 8.2 新增工作台 Header

建议新增以下 Header 之一：

#### 推荐字段
- `X-Workspace-Code`

#### 可选辅助字段
- `X-Root-Org-Type`
- `X-Selected-Identity-Id`

#### 示例
- `X-Workspace-Code: RS`
- `X-Workspace-Code: RSO`
- `X-Workspace-Code: RDT`

### 8.3 应用内上下文扩展

建议在 `UserInfo` 中新增字段：

```java
private String workspaceCode;
private String rootOrgType;
private Integer selectedIdentityId;
```

说明：

- `workspaceCode`：当前会话选中的工作台
- `rootOrgType`：当前工作台关联的组织根节点类型
- `selectedIdentityId`：若未来身份粒度细到“同根节点下多个身份”，可作为扩展

## 9. 表结构建议

### 9.1 方案一：根节点直接增加工作台配置字段（简单方案）

适用于根节点和工作台是一对一关系的场景。

#### 组织根节点表新增字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `workspace_code` | varchar | 工作台编码 |
| `biz_channel_type` | varchar | 默认业务归类渠道 |
| `workspace_name` | varchar | 工作台名称 |
| `app_scope` | varchar | 允许访问的 App 范围 |

优点：
- 改造简单
- 查询直接
- 快速上线

缺点：
- 后续一棵根节点如果支持多个工作台时不灵活

### 9.2 方案二：独立根节点-工作台映射表（推荐方案）

更适合后续扩展。

#### 表：`crm_root_org_workspace_rel`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint | 主键 |
| `root_org_type` | varchar | 根节点类型，如 DIRECT/BRAND/MARKET |
| `root_org_id` | bigint | 根节点ID |
| `workspace_code` | varchar | 工作台编码 |
| `workspace_name` | varchar | 工作台名称 |
| `biz_channel_type` | varchar | 默认业务归类渠道 |
| `app_scope` | varchar | 可展示入口范围，如 RS_APP,RD_APP |
| `is_default` | tinyint | 是否默认 |
| `status` | tinyint | 是否启用 |
| `created_by` | bigint | 创建人 |
| `created_at` | datetime | 创建时间 |
| `updated_by` | bigint | 更新人 |
| `updated_at` | datetime | 更新时间 |

优点：
- 扩展性更强
- 可支持同一根节点多个工作台
- 可支持不同 App 下展示策略配置
- 可支持灰度/启停

推荐采用此方案。

### 9.3 用户工作台偏好表

用于记录“上次退出时选择的工作台”。

#### 表：`crm_user_workspace_preference`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint | 主键 |
| `user_id` | bigint | 用户ID |
| `workspace_code` | varchar | 上次工作台 |
| `root_org_type` | varchar | 上次对应根节点类型 |
| `last_selected_at` | datetime | 最后选择时间 |
| `status` | tinyint | 状态 |

作用：
- 默认推荐工作台
- 降低用户重复选择成本

## 10. 接口改造点

### 10.1 登录后身份识别接口

#### 接口目标
返回用户当前可进入的工作台列表。

#### 返回示例
```json
{
  "userId": 10001,
  "channelCode": "RS",
  "workspaceList": [
    {
      "workspaceCode": "RS",
      "workspaceName": "直销工作台",
      "rootOrgType": "DIRECT",
      "isDefault": true
    },
    {
      "workspaceCode": "RSO",
      "workspaceName": "品牌工作台",
      "rootOrgType": "BRAND",
      "isDefault": false
    }
  ]
}
```

#### 用途
- 单身份直接进入
- 多身份展示选择页

### 10.2 工作台切换接口

#### 接口目标
用户切换工作台后，刷新当前会话上下文。

#### 请求示例
```json
{
  "workspaceCode": "RSO",
  "rootOrgType": "BRAND"
}
```

#### 处理动作
- 校验用户是否具备该工作台权限
- 更新会话上下文
- 刷新用户偏好记录
- 返回新的菜单/首页基础配置

### 10.3 菜单获取接口

菜单不再只看 `channelCode`，而要增加 `workspaceCode`。

#### 菜单加载规则
- `channelCode`：判断来源端和兼容逻辑
- `workspaceCode`：决定具体工作台菜单
- `role` + `groupScope`：决定是否显示某些子模块

### 10.4 工作盘点/排行榜/业绩查询接口

这些接口需要增加 `workspaceCode` 入参或上下文解析逻辑，作为统计口径分支条件。

#### 影响点
- 工作盘点指标集
- 排行榜指标集
- 业绩查询范围
- 产品报表口径
- 门店排序口径

### 10.5 门店、订单、售后接口

用户当前工作台用于决定：
- 展示内容
- 可操作项
- 数据范围口径

但对象本身的归类字段仍应由 `bizChannelType` 单独计算，不应直接使用 `workspaceCode` 覆盖。

---

## 11. 业务对象渠道归类建议

### 11.1 门店渠道类型

保持门店渠道类型作为**业务归类口径**独立计算，不使用 `channelCode` 或 `workspaceCode` 直接代替。

可继续沿用现有规则：
1. 若当前是企拍门店 → 企拍
2. 若关联销配体系人员 → 销配
3. 若关联直销体系人员 → 直销
4. 若关联品牌体系人员 → 直销
5. 若关联市场体系人员 → 销配
6. 若关联销配体系组织节点 → 销配
7. 若都不满足且门店类型=B端门店 → 直销

### 11.2 订单/售后/其它单据

建议统一为：
- 订单：按创单时刻用户的业务归类渠道
- 售后单：按售后单创建时刻用户的业务归类渠道
- 其它单据：按关联主单的业务归类渠道

注意：
这里用的是 **业务归类渠道**，不是入口 `channelCode`，也不是当前 `workspaceCode`。

---

## 12. 推荐落地步骤

### 12.1 第一阶段：先解耦概念，不动老逻辑主体
- 保留 `channelCode`
- 新增 `workspaceCode`
- 登录后新增工作台识别接口
- 多身份场景支持选择页
- 菜单与首页先按 `workspaceCode` 切换

### 12.2 第二阶段：扩展到权限与数据视图
- 门店列表按 `workspaceCode` 切换口径
- 工作盘点 / 排行榜 / 业绩查询按 `workspaceCode` 切换口径
- 完善身份切换后的上下文透传

### 12.3 第三阶段：收敛跨系统对象口径
- 门店渠道类型、订单渠道类型、售后单类型统一用 `bizChannelType`
- 数据平台、OMS、客服、商家等下游统一消费对象归类字段

### 12.4 第四阶段：支撑统一 App 演进
- 若后续 RS / RP / RD 合并为统一渠道 App，仅保留统一入口 `clientId`
- 但继续沿用 `workspaceCode` 控制工作台
- 网关侧可逐步弱化 `channelCode` 的业务意义，仅保留兼容价值

---

## 13. 风险与注意事项

### 13.1 风险一：旧逻辑仍按 `channelCode` 取菜单或判权限
需要全面排查：
- 菜单接口
- 首页接口
- 鉴权注解
- 数据范围拦截器
- 下游 Feign 透传逻辑

### 13.2 风险二：用户多组织归属导致工作台识别不稳定
需要明确：
- 哪些节点归属才算有效身份
- 同根节点多身份如何收敛
- 根节点缺失或异常时如何兜底

### 13.3 风险三：对象归类与用户当前工作台不一致带来理解偏差
需要在设计文档中明确说明：
- 工作台是“当前操作视图”
- 业务归类渠道是“对象统计归口”
- 二者不要求完全一致

### 13.4 风险四：Header 增加后透传链路遗漏
需要同步改造：
- 网关
- UserInfoInterceptor
- UserInfo
- Feign 透传拦截器
- 下游服务解析逻辑

---

## 14. 方案结论

本方案建议：

1. **保留 `channelCode`，继续作为请求入口通道标识**
2. **新增 `workspaceCode`，作为工作台路由、菜单加载、权限与指标切换的核心标识**
3. **保留 `bizChannelType` 作为门店/订单/售后等业务对象的归类标识**
4. **工作台匹配以组织树根节点为准，不再依赖 `channelCode`**
5. **建议通过“根节点-工作台映射表”实现配置化管理**
6. **优先完成登录路由、多身份选择、工作台上下文透传，再逐步扩展到报表与跨系统对象口径**

一句话概括：

> 将“入口通道”“当前工作台”“业务对象渠道归类”三类概念彻底拆开，分别建模、分别使用，以组织根节点驱动工作台路由，以业务规则驱动对象归类，在兼容现有 CRM 网关体系的前提下，为后续多角色扩展与统一 App 演进奠定基础。
